## Interaction Contract

### FE-01: Character Detail Link

**Trigger:** User sees a planned character with `characterId` in the editable list.
**Display:** `ExternalLink` icon (Lucide, `w-3.5 h-3.5`) appears to the right of the character name input. Only visible when `char.characterId` is truthy.
**Click behavior:** Emit `openCharacter` event with `char.characterId` to parent (`CreationCenter.vue`).
**Parent responsibility:** `CreationCenter.vue` must host a `CharacterDrawer` instance and handle the event to open it. CharacterDrawer uses Teleport-to-body so it overlays correctly.
**State after click:** CharacterDrawer opens with the selected character loaded.

### FE-02: Comparison View

**Trigger:** User opens ChapterPlanDrawer and switches to the "角色规划" tab, when `currentChapterPlan.chapterId` exists (chapter has been generated).

**Load sequence:**
1. User switches to "character" tab
2. If `currentChapterPlan.chapterId` exists, call `getChapterCharacters(projectId, chapterId)` to fetch actual characters
3. Run frontend comparison logic (ID-first + name-fallback matching)
4. Display comparison region

**Layout:**
- Comparison region sits above the existing editable character list
- Summary bar (collapsible header): shows match summary text + ChevronUp/Down toggle icon
- Below summary bar (when expanded): left-right grid (`grid-cols-2 gap-3`)
  - Left column: "规划角色" label + list of all planned characters with match status
  - Right column: "实际登场" label + list of all actual characters with match status
- Each character row: status icon + character name + role badge (if available)
- Content is compact -- name and badge only, no description in comparison view

**Matching logic:**
- For each planned character:
  - If `planned.characterId` exists: match by ID (`actual.characterId === Number(planned.characterId)`)
  - Else: match by exact name (`actual.characterName === planned.characterName`)
- Three result groups:
  - `matched`: appears in both planned and actual (green `CheckCircle2`)
  - `plannedOnly`: in planned but not actual (red `XCircle`)
  - `actualOnly`: in actual but not planned (amber `AlertTriangle`)

**Collapse/Expand:**
- Default state: expanded (CONTEXT.md D-03)
- Click summary bar to toggle
- Transition: simple `v-if` toggle, no animation required (lightweight)
- When collapsed: only summary bar visible
- When expanded: summary bar + detail grid

**Visibility rules:**
- Hidden when `currentChapterPlan.chapterId` is null/undefined (chapter not generated)
- Hidden when chapter is currently generating
- Shown when chapter has been generated, even if actual characters list is empty (shows all planned as "plannedOnly")

### Responsive Handling

Drawer width is fixed at 640px (`w-[640px]`). With `grid-cols-2 gap-3` and `p-6` side padding, each panel gets approximately 280px. This is sufficient for name + badge display. No responsive breakpoint needed within the drawer.

## Pitfall Guards

These are known risks from RESEARCH.md that the executor must handle:

1. **chapterId null guard:** Check `currentChapterPlan.chapterId` before calling API. If null, hide comparison region entirely. (D-04)
2. **CharacterDrawer hosting:** `CreationCenter.vue` does not currently import CharacterDrawer. Executor must add CharacterDrawer instance to CreationCenter.vue and wire the `openCharacter` event from ChapterPlanDrawer. CharacterDrawer uses Teleport-to-body so no layout conflict.
3. **plannedCharacters re-parsing:** Reuse existing `editableCharacters` ref for comparison, do not create a second parse of `form.plannedCharacters`.
4. **Empty actual characters:** Handle gracefully -- all planned characters show as "plannedOnly" (red). Do not assume actual list is non-empty.
5. **Drawer width constraint:** Keep comparison rows compact (name + badge only). No descriptions in comparison view.
