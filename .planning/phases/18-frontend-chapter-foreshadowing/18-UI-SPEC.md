## Interaction Specification

### Data Loading

On mount (`onMounted`), ForeshadowingTab makes up to 3 API calls:

1. **Planted in this chapter:** `GET /api/novel/{projectId}/foreshadowings?plantedChapter={chapterNumber}`
2. **Callback in this chapter:** `GET /api/novel/{projectId}/foreshadowings?plannedCallbackChapter={chapterNumber}`
3. **Volume active (optional):** `GET /api/novel/{projectId}/foreshadowings?plantedVolume={volumeNumber}` (only if `volumeNumber` prop is available), then client-side filter to exclude items already in queries 1-2.

**CRITICAL NOTE -- Backend Gap:** The current `ForeshadowingController.getForeshadowingList()` only accepts 4 `@RequestParam` fields: `type`, `layoutType`, `status`, `currentChapter`. The `plantedChapter`, `plannedCallbackChapter`, `plantedVolume`, and `plannedCallbackVolume` fields exist in `ForeshadowingQueryDto` and are handled by `ForeshadowingService.getForeshadowingList()`, but the controller does NOT bind them from request parameters. The controller explicitly constructs the DTO and only sets those 4 fields (controller lines 74-79).

**Resolution:** Either:
- (A) Add `@RequestParam(required=false)` parameters for `plantedChapter`, `plannedCallbackChapter`, `plantedVolume` to the controller method, OR
- (B) Use `currentChapter` parameter for query 2 (it finds foreshadowing where `plannedCallbackChapter <= currentChapter` with pending/in_progress status), and fetch all foreshadowing for the project then filter client-side for query 1.

**Recommended approach:** (A) -- adding the missing request params is a 4-line change in the controller and requires no service or mapper changes. This is cleaner than client-side filtering. The planner should include this as a task.

### Loading State

- Show skeleton placeholders (3 cards) while loading.
- Skeleton card: `animate-pulse bg-gray-200 dark:bg-gray-700 rounded-lg h-20`

### Modal Form (Add/Edit)

**Trigger:** "添加伏笔" button (add mode) or pencil icon on card (edit mode).

**Layout:**
```
+---------------------------------------------------+
| 添加伏笔 / 编辑伏笔                       [X]      |
+---------------------------------------------------+
| 伏笔标题 *                                        |
| [input text]                                       |
|                                                    |
| 伏笔类型 *  布局线类型                              |
| [select]    [select]                               |
|                                                    |
| 伏笔描述                                           |
| [textarea rows=4]                                  |
|                                                    |
| 埋设章节      计划回收分卷                           |
| [input num]   [input num]                          |
| (disabled in edit mode)                            |
|                                                    |
| 计划回收章节  优先级                                 |
| [input num]   [select 1-5]                         |
|                                                    |
| 备注                                               |
| [textarea rows=2]                                  |
+---------------------------------------------------+
|                              [取消]  [保存]        |
+---------------------------------------------------+
```

**Modal z-index:** `z-[60]` (above the drawer's `z-50`). Uses `<Teleport to="body">`.

**Field specifications:**
- Title: required, text input, placeholder "输入伏笔标题"
- Type: required, select with options: character("人物"), item("物品"), event("事件"), secret("秘密")
- LayoutType: optional, select with options: bright1("明线1"), bright2("明线2"), bright3("明线3"), dark("暗线")
- Description: optional, textarea, placeholder "描述伏笔内容..."
- PlantedChapter: number input, pre-filled with current chapter number. Disabled when editing (per D-05)
- PlannedCallbackVolume: optional number input
- PlannedCallbackChapter: optional number input
- Priority: select 1-5, default 3
- Notes: optional textarea

**Validation:**
- Title is required. Show inline error "请输入伏笔标题" below the field.
- PlantedChapter is required for new foreshadowing. Pre-filled from current chapter.
- No frontend distance validation -- the backend handles this (`MIN_FORESHADOWING_DISTANCE = 3`).

### Delete Flow

1. User clicks trash icon on card.
2. `ConfirmDialog` opens with:
   - title: "删除伏笔"
   - message: "确定要删除伏笔「{title}」吗？此操作无法撤销。"
   - variant: "danger"
   - confirmText: "删除"
3. On confirm: call `DELETE /api/novel/{projectId}/foreshadowings/{id}`
4. On success: toast "伏笔已删除", reload data.
5. On error: toast the error message.

### Collapsible Sections

- Each section header is clickable to toggle collapse.
- State tracked via refs: `plantSectionExpanded`, `callbackSectionExpanded`, `volumeSectionExpanded`.
- "待埋设" and "待回收" default to expanded.
- "分卷伏笔参考" defaults to collapsed.
- Animation: none (instant toggle, matching existing `v-if` pattern in ChapterPlanDrawer).

## Edge Cases and Error Handling

| Scenario | Behavior |
|----------|----------|
| volumeNumber not available | Skip the "分卷伏笔参考" section entirely. Do not render it. |
| API call fails | Show error toast. Display empty state with error text. Do not leave skeleton loading forever. |
| Foreshadowing with no plannedCallbackChapter | Show card with "第N章埋设" only (no arrow). Badge shows status. |
| Foreshadowing status is "completed" | Show in the relevant section with green "已完成" badge. Card uses slightly reduced opacity (`opacity-75`). |
| Creating foreshadowing with distance validation error | Backend returns error. Show toast with backend error message. Keep modal open so user can correct. |
| Concurrent edits (user edits while AI updates status) | Last write wins. Reload data after any CRUD operation to get latest state. |
| Modal z-index conflict with drawer | Modal uses `z-[60]` via Teleport to body. ConfirmDialog already uses `z-50` with Teleport -- it may need `z-[60]` as well when triggered from within ForeshadowingTab. |
