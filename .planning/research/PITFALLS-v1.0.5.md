# Domain Pitfalls: Chapter Character Planning (v1.0.5)

**Domain:** Adding character planning to chapter outline generation in an AI novel system
**Researched:** 2026-04-07
**Confidence:** HIGH (derived from direct codebase analysis of existing chapter planning, character extraction, XML parsing, and prompt building patterns)

---

## Critical Pitfalls

### Pitfall 1: Planned Characters != Actually Generated Characters -- The Core Drift Problem

**What goes wrong:**
The chapter plan says "Li Yun confronts Elder Wang, showing anger and determination." But when the chapter is actually generated, the LLM writes a scene where Li Yun is calm and strategic, barely confronts Elder Wang, and instead focuses on a side character. The planned character roles, emotions, and even presence are not followed. This is the single most important risk of this entire milestone because the value proposition is "plan characters -> generate by plan -> extract and verify," and the middle step is the weakest link.

**Why it happens:**
LLMs treat prompts as strong suggestions, not contracts. The current chapter generation prompt (in `PromptTemplateBuilder.buildChapterPrompt`) already injects chapter plan data as `chapterOutline`, `startingScene`, `endingScene`. But the prompt does NOT say "you MUST include these exact characters doing these exact things." It says "follow the outline" which LLMs interpret loosely. Additionally, the current `characterInfo` variable injects ALL non-NPC characters from the entire project (see `buildCharacterPromptInfoList` -- it calls `getNonNpcCharacters(projectId)` with no chapter filter). The LLM sees 20 characters and freely picks from the full set, ignoring which ones the plan specifically chose.

The drift is amplified by:
1. The chapter plan prompt and the chapter generation prompt are different templates (`llm_outline_chapter_generate` vs `llm_chapter_generate_standard`) with different instructions.
2. The generation prompt's `characterInfo` block currently shows ALL characters, not the planned subset.
3. There is no "constraint language" in the generation prompt saying "the following characters MUST appear in this chapter."

**Consequences:**
- Extracted character data (via `ChapterCharacterExtractService`) does not match planned data, making the "verify" step unreliable.
- Users see planned characters in the outline but get different characters in the actual chapter.
- The whole planning system loses credibility -- why plan if the LLM ignores it?

**Prevention:**
1. In the generation prompt, inject planned characters as a MUST-FOLLOW constraint block, not background information. Format: "The following characters MUST appear in this chapter: [list]. Their roles: [descriptions]. Do NOT add unplanned major characters."
2. Change `buildCharacterPromptInfoList` to accept a planned character filter. When a chapter plan has `planned_characters`, use ONLY those characters in the prompt, not all project characters.
3. Add a "strictness" instruction: "Character appearances must match the planned cast. If a planned character has no scene, assign them a brief moment. Do not introduce new named characters without planning justification."
4. After extraction, compare planned vs actual character lists and log discrepancies at WARN level.

**Detection:**
- Extracted character names that were NOT in the plan.
- Planned characters that do NOT appear in the extracted list.
- Character count mismatch: plan says 3 characters, extraction finds 5.

**Phase to address:**
Phase 1 (chapter plan template upgrade) designs the XML structure; Phase 2 (generation prompt injection) must enforce the plan. Both phases must be tested together before Phase 3 (extraction verification).

### Pitfall 2: AI-Generated Character Names Do Not Match Database Names

**What goes wrong:**
The chapter plan LLM outputs `<character><name>Li Yun</name></character>`. But in the `novel_character` table, the character is stored as "Li Yun (Jr.)" or "Li Yunxiao" or "Little Li Yun." The backend tries to match by name to inject character details (personality, background, last appearance) into the generation prompt. The match fails, and the LLM generates the chapter with no character context -- it makes up personality traits, ignores previous development, and creates inconsistency.

This is the EXACT same problem already solved for factions (three-tier name matching in `WorldviewXmlParser`). But the current chapter-level character matching in `ChapterCharacterExtractService.isSameCharacter()` only does exact match + gender check. It does NOT do fuzzy matching.

**Why it happens:**
The chapter plan is generated by a different LLM call (`llm_outline_chapter_generate`) than the character extraction call (`llm_chapter_character_extract`). The plan LLM has no character name reference list -- it inventes names based on the story context. Even if you provide a character list in the plan prompt, the LLM may still use abbreviations, nicknames, or slight variations.

Looking at the existing code:
- `ChapterGenerationTaskStrategy.buildChapterPromptUsingTemplate()` at line 435 sets `variables.put("characterInfo", "暂无登场角色")` -- it provides NO character reference to the plan generation LLM.
- `ChapterCharacterExtractService.isSameCharacter()` at line 489 requires `existing.getName().equals(charDto.getName())` -- exact match only.
- The three-tier matching in `WorldviewXmlParser` (exact -> strip suffix -> contains) is NOT reused.

**Consequences:**
- Planned characters cannot be resolved to database IDs.
- Character background, personality, and last-appearance state are not injected into the generation prompt.
- Each chapter treats characters as if they are appearing for the first time.

**Prevention:**
1. **In the chapter plan prompt**, inject the existing character name list as a mandatory reference (same pattern as `ChapterCharacterExtractService.buildExistingCharactersText`). Use explicit instruction: "Use ONLY names from the existing character list. If a new character is needed, mark them with `[NEW]` prefix."
2. **In the plan XML parser**, reuse the three-tier name matching from `WorldviewXmlParser` (exact -> strip suffix -> contains). Extract it into a shared utility or call the existing matcher.
3. **Handle [NEW] characters gracefully**: If a planned character name has no match in the database, mark it in the plan as "new character" and skip detail injection (the LLM will create them fresh).
4. **Log mismatches**: Every failed name match should produce a WARN log with both the planned name and the list of candidates.

**Detection:**
- WARN logs during plan parsing showing "name match failed."
- Generation prompts that have no character detail blocks despite having planned characters.
- Chapters where a known character acts out of character (because the LLM had no reference).

**Phase to address:**
Phase 1 (plan template upgrade) must include character name injection in the prompt. Phase 2 (plan XML parsing) must use fuzzy name matching.

### Pitfall 3: planned_characters JSON Structure Too Rigid or Too Loose

**What goes wrong:**
The `novel_chapter_plan` table will get a `planned_characters` JSON column (the table already has a `character_arcs` JSON column). If the JSON schema is too rigid (fixed required fields), the AI will frequently produce malformed JSON that fails validation, and the plan is lost entirely. If the schema is too loose (everything optional), the data becomes useless noise -- you get `{"name": "Li Yun"}` with no role or scene information, providing no value over not having the field.

**Why it happens:**
Looking at how `key_events` works in the current system -- it is stored as a `text` column and sometimes contains JSON (5-stage format: `{opening: [...], development: [...]}`), sometimes plain text. The `PromptTemplateBuilder` at line 534 tries `objectMapper.readTree()` with a catch fallback to raw text. This dual-format pattern works but is fragile.

For `planned_characters`, the temptation is to design a rich schema:
```json
[{
  "characterName": "Li Yun",
  "characterId": 5,
  "roleInChapter": "protagonist",
  "sceneDescription": "Confronts Elder Wang",
  "emotionArc": "anger -> determination -> resolution",
  "keyDialogue": "I will not back down!",
  "interactions": ["Elder Wang", "Sister Chen"]
}]
```
But the LLM will inconsistently populate these fields. `characterId` will be wrong (LLM does not know IDs). `emotionArc` may be missing. `interactions` may reference characters not in the plan.

**Consequences:**
- Parsing fails on malformed JSON -> planned characters lost -> generation prompt has no character plan.
- Partial data (name only) provides no actionable constraint -> LLM ignores it.
- Over-complex schema makes the plan prompt longer, consuming tokens that could be used for story content.

**Prevention:**
1. **Keep the schema minimal**: `name` (required), `roleType` (required, one of: protagonist/supporting/antagonist/npc), `sceneBrief` (optional, max 50 chars). Three fields, nothing more. The detailed character behavior should live in the `plotOutline` text, not in structured JSON.
2. **Store as XML during AI generation, parse to JSON for database**: Follow the existing pattern -- AI outputs XML tags inside the chapter plan XML, backend parses and converts to JSON for storage. This avoids asking the LLM to produce valid JSON inside XML.
3. **Use Jackson `ObjectMapper` with `FAIL_ON_UNKNOWN_PROPERTIES = false`** (already configured in `XmlParser`) so extra fields do not break parsing.
4. **Validate after parse, not during**: Parse generously, then validate that each entry has at least `name`. Drop entries without `name` with a WARN log.

**Detection:**
- Planned characters JSON column is null or empty after plan generation.
- JSON parsing exceptions in logs during plan save.
- Chapters with planned_characters but no matching names in the character table.

**Phase to address:**
Phase 1 (plan template upgrade + XML structure design). The schema must be decided before writing the template or the parser.

### Pitfall 4: Chapter Plan XML Structure Change Breaks Existing Parser

**What goes wrong:**
The chapter plan XML currently uses this structure (defined in `ChapterPlanXmlDto` and `common.xml.ChapterPlanXmlDto`):
```xml
<c>
  <o>
    <n>1</n>
    <t>title</t>
    <p>plot</p>
    <e>events</e>
    <g>goal</g>
    <w>3000</w>
    <s>start scene</s>
    <f>end scene</f>
  </o>
</c>
```
Adding character planning requires new tags inside `<o>`:
```xml
<o>
  ...existing tags...
  <chars>
    <ch><name>Li Yun</name><role>protagonist</role><scene>Brief scene desc</scene></ch>
  </chars>
</o>
```
But the XML is parsed by `ChapterGenerationTaskStrategy.parseChaptersXml()` which uses `XmlParser.parse(xmlStr, ChapterPlanXmlDto.class)`. This Jackson XML parsing maps fixed `@JacksonXmlProperty` annotations. Adding new tags means updating the DTO, which means updating ALL consumers of the DTO.

Worse: there are TWO `ChapterPlanXmlDto` classes in the codebase:
1. `com.aifactory.common.xml.ChapterPlanXmlDto` (the older one)
2. `com.aifactory.dto.ChapterPlanXmlDto` (the newer one with Schema annotations)

The `ChapterGenerationTaskStrategy` imports from `com.aifactory.dto.ChapterPlanXmlDto`. If you update one and not the other, parsing breaks silently.

**Why it happens:**
The system evolved with duplicated DTOs. The `common.xml` package has the original, the `dto` package has the Swagger-documented version. Both have `ChapterPlanItem` inner classes. The XML tag `<f>` maps to `chapterEndingScene` in one and might conflict with a new `<chars>` sub-element.

**Consequences:**
- Parsing existing chapter plans (without character tags) still works, but new plans with character tags may fail if the parser is not updated.
- Updating one DTO but not the other causes inconsistent behavior across different code paths.
- The Jackson XML parser throws on unexpected nested elements if not properly annotated.

**Prevention:**
1. **Consolidate to a single `ChapterPlanXmlDto`** before adding character tags. Delete or deprecate the duplicate. Use `@JacksonXmlElementWrapper(useWrapping = false)` and `@JacksonXmlProperty(localName = "ch")` for the new character list.
2. **Make the new character tags fully optional**: Use `List<PlannedCharacter>` which defaults to null. The parser should handle `<o>` elements with and without `<chars>` blocks.
3. **Test backward compatibility**: Parse the old XML format (without `<chars>`) and verify it produces valid `ChapterPlanItem` objects with null character list.
4. **Update both parsers if keeping both DTOs**: The `ChapterGenerationTaskStrategy.parseChaptersXml()` and any other consumer must all be updated.

**Detection:**
- `XmlParseException` in logs during chapter plan generation.
- Parsed chapter plans have null `plannedCharacters` despite the XML containing character data.
- Old chapter plans (generated before this change) fail to parse.

**Phase to address:**
Phase 1 (plan template + XML parsing). The DTO consolidation must happen as the first sub-task before any other changes.

### Pitfall 5: Generation Prompt Injects Planned Characters But Also All Project Characters -- Contradictory Signals

**What goes wrong:**
The current `PromptTemplateBuilder.buildTemplateVariables()` at line 204 calls `buildCharacterPromptInfoList(projectId, chapterPlan.getChapterNumber())` which loads ALL non-NPC characters in the project. This is injected as the `characterInfo` template variable. If you also inject `plannedCharacters` (the subset for this chapter), the LLM receives TWO character lists:
1. "Here are ALL characters in the project" (from `characterInfo`)
2. "Here are the characters PLANNED for this chapter" (from the new `plannedCharacters` block)

The LLM gets confused by the contradiction. It sees "Sister Chen" in the full list but not in the plan. Should it include Sister Chen? The full list implies she exists and might be relevant. The plan list implies she should not appear. The LLM usually follows the longer, more detailed list (the full project characters) and ignores the shorter planned list.

**Why it happens:**
The `characterInfo` variable was designed for a world without chapter-level character planning. It provides context ("here is who exists") rather than constraint ("here is who must appear"). The new `plannedCharacters` provides constraint. But both are present simultaneously, and the LLM cannot distinguish context from constraint.

**Consequences:**
- Planned characters are overshadowed by the full character list.
- The LLM includes unplanned characters because they appear in the `characterInfo` block.
- The planning system appears to have no effect.

**Prevention:**
1. **When `plannedCharacters` exists, replace `characterInfo` with ONLY the planned characters' details**. Do not show the full project character list.
2. **When `plannedCharacters` is null/empty** (old plans, or plans generated before this feature), fall back to the current behavior (show all project characters).
3. **Add a `hasPlannedCharacters` boolean** in the template variables to let the template choose which block to render.
4. **In the prompt template**, use conditional text: "if {{hasPlannedCharacters}} == true, show planned character constraint; else show full character context."

**Detection:**
- Generation prompts that contain both a "planned characters" block and a "all project characters" block.
- LLM output that includes characters not in the plan.
- A/B test: chapters with planned-only injection vs chapters with both lists, comparing character adherence.

**Phase to address:**
Phase 2 (generation prompt injection). Must be addressed when modifying the template and `PromptTemplateBuilder`.

### Pitfall 6: Planned Character Data Stale After User Edits

**What goes wrong:**
The user generates a chapter plan with character assignments. Then the user edits the plan -- changes the plot outline, removes a character from the planned list, adds a new one. But the `planned_characters` JSON in the database is not updated because the edit API does not know about the new field. The generation prompt uses the stale planned characters, producing a chapter that does not match the edited plan.

**Why it happens:**
Looking at the existing `ChapterUpdateRequest` and chapter plan update flow, the current edit operations target fields like `chapterTitle`, `plotOutline`, `keyEvents`. The new `planned_characters` field is not in the existing edit DTO. If it is added to the entity but not to the update API, the frontend cannot update it.

Additionally, even if the frontend CAN edit planned characters, the user might change the plot outline in a way that implicitly changes which characters should appear (e.g., removing a battle scene removes the need for the antagonist). But the `planned_characters` JSON still lists the antagonist.

**Consequences:**
- User edits are ignored during generation.
- Generated chapters follow the original plan, not the edited one.
- Trust in the system degrades.

**Prevention:**
1. **Add `planned_characters` to the chapter plan update API** (DTO + controller + service). Allow full replacement of the JSON.
2. **Frontend must send the full planned_characters array on every plan edit**, not partial updates. This avoids merge conflicts.
3. **Consider a "re-plan characters" button** that re-runs character planning for a single chapter after the user edits the plot outline, rather than requiring manual character list editing.
4. **On plan edit that changes `plotOutline`, show a warning**: "You changed the plot outline. The planned characters may no longer match. Consider re-planning characters."

**Detection:**
- `planned_characters` JSON in DB does not match what the frontend shows after edit.
- Generation prompt uses old character list after user confirmed a new one.
- API returns 200 on update but `planned_characters` is unchanged in the next GET.

**Phase to address:**
Phase 1 (database extension) adds the column; Phase 3 (frontend) must expose the edit capability. Both must be tested end-to-end.

## Moderate Pitfalls

### Pitfall 7: Jackson XML Cannot Parse Nested `<ch>` Inside `<o>` If `<ch>` Also Appears Elsewhere

**What goes wrong:**
The new XML structure adds `<ch>` (character) tags inside `<o>` (chapter) tags. But if the existing XML already has `<ch>` used elsewhere in the system (it does -- `ChapterPlanXmlDto` currently has no character tags, but the character extraction XML uses `<C>` for characters in `ChapterCharacterExtractXmlDto`). If you accidentally use a tag name that conflicts with another XML format, the parser may misinterpret elements.

More concretely: if you use `<ch>` as the character tag, and the LLM accidentally outputs a `<ch>` tag at the wrong nesting level (outside an `<o>` tag), Jackson XML will try to parse it as a chapter plan character and fail.

**Why it happens:**
Single-letter or short tag names (`<c>`, `<o>`, `<p>`, `<s>`, `<f>`) are already heavily used in the chapter plan XML. Adding more short tags increases collision risk. The LLM also sometimes produces malformed XML where tags appear at wrong nesting levels.

**Prevention:**
1. Use a distinctive tag name for planned characters that cannot be confused with existing tags. Recommendation: `<chars>` for the wrapper and `<pc>` (planned character) for each entry. The `pc` prefix makes it unique.
2. Use `@JacksonXmlElementWrapper(useWrapping = false)` with careful annotation to avoid Jackson wrapping the list in an extra element.
3. Add null-checking and try-catch around the character list parsing so a malformed character entry does not break the entire chapter plan parsing.

**Phase to address:**
Phase 1 (XML structure design).

### Pitfall 8: planned_characters and character_arcs Overlap Confusion

**What goes wrong:**
The `novel_chapter_plan` table already has a `character_arcs` JSON column (see `init.sql` line 381: `"character_arcs" json DEFAULT NULL`). This column stores "character arc changes" per chapter -- presumably how characters evolve. The new `planned_characters` field stores "which characters appear and what they do." These two fields overlap conceptually: both contain per-character data for a chapter.

If both are populated, which one does the generation prompt use? If `character_arcs` contains "Li Yun gets angry" and `planned_characters` contains "Li Yun is calm and strategic," the LLM receives contradictory signals.

**Why it happens:**
The `character_arcs` column exists in the schema but the entity `NovelChapterPlan` does NOT have a `character_arcs` field (confirmed by reading the entity -- it only has fields up to `foreshadowingPayoff`). This means `character_arcs` was added to the schema but never implemented. However, the column exists in the database, and future code might try to use it.

**Prevention:**
1. **Decide NOW**: either `character_arcs` is the planned_characters field (repurpose the existing column), or it is a separate concept and should remain unused.
2. **Recommended**: repurpose `character_arcs` as the storage for planned characters, renaming the concept to "character arcs within this chapter" which naturally maps to "planned character roles and development in this chapter." This avoids adding yet another JSON column.
3. If repurposing, update the entity to add the field with a clear Javadoc explaining its dual purpose.
4. If not repurposing, add a NEW `planned_characters` column and explicitly mark `character_arcs` as unused in the entity comments.

**Phase to address:**
Phase 1 (database extension). Must be decided before writing the SQL migration.

### Pitfall 9: Token Budget Exceeded by Adding Character Planning to Chapter Plan Prompt

**What goes wrong:**
The chapter plan prompt (`llm_outline_chapter_generate`) already asks the LLM to generate 5 chapters per batch with 8 XML fields each. Adding character planning (planned characters per chapter) means each `<o>` block gets 1-4 more `<pc>` child elements with 3 fields each. For a 5-chapter batch with 3 characters per chapter, that is 45 additional XML elements. The LLM's context window may not have enough room, or the output gets truncated mid-XML, causing parsing failure.

**Why it happens:**
The current batch size is 5 chapters (line 219 in `ChapterGenerationTaskStrategy`). Each chapter currently has 8 XML fields. With character planning adding ~9 more fields per chapter (3 characters x 3 fields), the output grows by ~50%. If the LLM was already near its output token limit, it will be truncated.

**Prevention:**
1. **Reduce batch size to 3 chapters** when character planning is enabled. This is a simple, safe change.
2. **Use very concise XML tags**: `<pc><n>Name</n><r>role</r><b>brief scene</b></pc>` -- 3 fields, single-letter tags.
3. **Monitor output length**: after generation, check if the XML is complete (closing `</c>` tag exists). If truncated, log a warning and retry with fewer chapters.
4. **Make character planning optional per generation**: allow the user to toggle character planning on/off. When off, use the current 5-chapter batch with no character fields.

**Phase to address:**
Phase 1 (template design). Token budget must be estimated before finalizing the XML structure.

### Pitfall 10: Extraction After Generation Creates False Negatives

**What goes wrong:**
After chapter generation, `ChapterCharacterExtractService.extractCharacters()` runs to identify which characters actually appeared. But the extraction LLM may miss characters that the LLM mentioned only briefly or by pronoun. The extraction says "2 characters found" but the plan said "4 characters." This looks like a drift failure, but it is actually an extraction failure. The character DID appear in the generated text, but the extraction LLM did not detect them.

**Why it happens:**
Looking at `ChapterCharacterExtractService`, it uses `SKIP_WORDS` to filter out pronouns. But a character who appears in only one sentence ("Li Yun nodded to Chen.") may not be extracted because the LLM decides "Chen" is too minor to report. The extraction prompt asks for "all characters" but LLMs naturally prioritize characters with more screen time.

This creates a false negative in the planned-vs-actual comparison, making the system appear less reliable than it actually is.

**Prevention:**
1. **In the extraction prompt, explicitly list the planned character names**: "The following characters are expected to appear in this chapter: [names]. You MUST report them even if they only appear briefly."
2. **Treat extraction misses as soft warnings, not hard failures**. Log as INFO, not WARN, unless a planned protagonist is completely absent.
3. **Consider a simpler verification method**: instead of re-extracting characters from the text, just search for planned character names in the generated content (string contains check). This is less sophisticated but more reliable for the "did they appear at all?" question.
4. **Post-extraction, cross-reference**: if a planned character name appears in the generated text but was not extracted, flag it as "extraction gap" rather than "planning gap."

**Phase to address:**
Phase 3 (extraction + verification). The verification logic must distinguish extraction failure from planning failure.

## Minor Pitfalls

### Pitfall 11: Frontend Shows Planned Characters as Static Text, Users Cannot Reorder or Adjust

**What goes wrong:**
The frontend displays planned characters as a read-only list. Users see "Li Yun - protagonist - confronts Elder Wang" but cannot drag to reorder, adjust the brief description, or change the role type. The only way to modify is to regenerate the entire chapter plan, which is heavy-handed.

**Prevention:**
Provide inline editing of planned character entries: editable role type dropdown, editable brief description field. Add/remove buttons for adding unplanned characters or removing planned ones.

**Phase to address:**
Phase 4 (frontend display). Must be part of the initial UI design, not added later.

### Pitfall 12: planned_characters Empty for Chapters Generated Before This Feature

**What goes wrong:**
All existing chapter plans (generated before v1.0.5) have null `planned_characters`. The generation prompt falls back to showing all project characters (current behavior). But the extraction service now tries to compare planned vs actual and logs warnings about "no planned characters found" for every pre-existing chapter, flooding the logs.

**Prevention:**
Guard the comparison: only run planned-vs-actual verification when `planned_characters` is not null. Log at DEBUG level, not WARN, for chapters without planned characters.

**Phase to address:**
Phase 2 (generation prompt). The fallback path must be tested with null planned_characters.

### Pitfall 13: Chapter Plan Template Version Collision

**What goes wrong:**
The `llm_outline_chapter_generate` template (already in the database) will be updated to include character planning instructions. But existing projects may have cached the old template version. If the template versioning system (`ai_prompt_template_version`) is not properly handled, some projects get the old template (no character output) and others get the new template. Inconsistent behavior.

**Prevention:**
Follow the existing template versioning pattern: create a new version row, activate it, and deactivate the old version. Do NOT update the existing version in place. The `PromptTemplateService.executeTemplate()` resolves the active version by template code, so this should work seamlessly.

**Phase to address:**
Phase 1 (template upgrade). Use the established versioning workflow.

## Technical Debt to Avoid

| Shortcut | Immediate Benefit | Long-term Cost | Verdict |
|----------|-------------------|----------------|---------|
| Store planned_characters as plain text, not JSON | No parsing code needed | Cannot query/filter by character, no structured comparison | REJECT -- use JSON |
| Show all project characters alongside planned characters | No need to change PromptTemplateBuilder | Contradictory signals to LLM (Pitfall 5) | REJECT -- planned-only when available |
| Add planned_characters as a new column, ignore character_arcs | Clean separation | Two overlapping JSON columns causing confusion forever | RISKY -- repurpose character_arcs or explicitly deprecate |
| Update only one of the two ChapterPlanXmlDto classes | Less work, faster | Silent parsing failure in the other code path (Pitfall 4) | REJECT -- consolidate first |
| Skip name matching, trust LLM to use exact names | Simpler parsing | High failure rate in practice (proven by faction experience) | REJECT -- always use fuzzy matching |

## "Looks Done But Is Not" Checklist

- [ ] **Planned characters in plan XML**: Plan XML contains `<pc>` tags, parser extracts them -- BUT generation prompt does not inject them (only storage, no usage)
- [ ] **Generation prompt uses planned characters**: Prompt contains planned character block -- BUT also contains all project characters (contradictory signals)
- [ ] **Name matching works**: Parser matches planned "Li Yun" to DB "Li Yun" -- BUT fails on "Little Li Yun" or "Li Yun (Jr.)" (no fuzzy matching)
- [ ] **Frontend shows planned characters**: UI displays character list -- BUT users cannot edit it (read-only)
- [ ] **Backward compatibility**: Old plans parse correctly -- BUT old plans have null planned_characters and the code does not handle null
- [ ] **Extraction verification**: Post-generation extraction runs -- BUT compares against stale planned_characters (not the latest user-edited version)
- [ ] **Template versioning**: New template version created -- BUT old version still active for some projects
- [ ] **Both ChapterPlanXmlDto classes updated**: One class has character fields -- BUT the other (used by a different code path) does not
